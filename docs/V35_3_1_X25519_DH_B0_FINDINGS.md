# v35.3.1 x25519_dh — B0 Pre-Flight Audit Findings

> The v35.3.0 release notes positioned x25519_dh as a small "future
> patch candidate" requiring x25519-dalek. **B0 audit surfaces that
> v35.3.0's x25519_generate is itself NOT real X25519 — the impl
> uses Ed25519 key derivation as a proxy.** Adding x25519_dh on
> top of broken keygen would compound the problem. This doc surfaces
> the issue + 3 honest scope options.

## §1 — The bug

`src/stdlib_v3/crypto.rs:541-561`:

```rust
/// Generate an X25519 key pair for key exchange.
pub fn x25519_generate() -> X25519KeyExchange {
    let secret = random_bytes(32);
    let mut secret_arr = [0u8; 32];
    secret_arr.copy_from_slice(&secret);
    // Clamp per X25519 spec
    secret_arr[0] &= 248;
    secret_arr[31] &= 127;
    secret_arr[31] |= 64;

    // Public key = secret * basepoint (simplified: just derive from secret)
    // For real X25519, use x25519-dalek. Here we use Ed25519 key derivation as proxy.
    use ed25519_dalek::SigningKey;
    let signing = SigningKey::from_bytes(&secret_arr);
    let public = signing.verifying_key().to_bytes();

    X25519KeyExchange {
        public_key: public,
        secret_key: secret_arr,
    }
}
```

**The "public key" returned is an Ed25519 public key** (Edwards
curve), NOT an X25519 public key (Montgomery curve). These are
DIFFERENT cryptographic objects:
- X25519: scalar multiplication on Curve25519 (Montgomery form)
- Ed25519: scalar multiplication on Edwards25519 (Twisted Edwards form)
- Conversion exists (Edwards → Montgomery via birational map) but
  is NOT what the current code does

**Consequence for v35.3.0 users:**
- `x25519_generate()` returns a (correctly-clamped) X25519 secret
  key + a wrongly-derived Ed25519 public key
- Any peer doing real X25519 (e.g., libsodium, x25519-dalek,
  OpenSSL) and computing `dh(their_secret, our_public)` would get
  a WRONG shared secret
- Any v35.3.0 user attempting interop with standard X25519
  implementations will silently get wrong results — no error, no
  crash, just incorrect key exchange

**Severity:** This is a documented-in-code-comment workaround that
was acknowledged as "not real X25519" but shipped anyway under the
`x25519_*` name. Honesty-of-API is broken; security implications
depend on whether anyone uses it for real key exchange yet.

## §2 — Three honest scope options for v35.3.1

### Option A — Fix x25519_generate + add x25519_dh (proper, ~1.5-2h)

- Add `x25519-dalek = "2"` to Cargo.toml
- Replace `x25519_generate` impl with `StaticSecret::random_from_rng(...)` + `PublicKey::from(&secret)`
- Add `x25519_dh(secret_hex, peer_pub_hex) -> shared_secret_hex` using `secret.diffie_hellman(&peer_public)`
- Update integration tests: add real round-trip (Alice generates → Bob generates → both compute DH → shared secrets equal)

**BREAKING for v35.3.0 keys:** any keys generated by v35.3.0's
`x25519_generate` will NOT work with v35.3.1's `x25519_dh` (because
the v35.3.0 "public key" is wrong format). Document explicitly.

This is the only correct path forward.

### Option B — Add x25519_dh that matches the broken impl (~30min, REJECTED)

Add x25519_dh that does ed25519-style scalar mult (so v35.3.0 keys
work with v35.3.1 dh). **This is unacceptable** — would be shipping
a security primitive labeled X25519 that is actually Ed25519-based.
Violates CLAUDE.md §6.6 "honesty upfront" + §6.9 research integrity.

### Option C — Document the bug, deprecate broken impl, defer fix to v36.x (~30min)

- Add deprecation notice to `x25519_generate` in `crypto.rs` doc + analyzer hint
- Add big warning to v35.3.0 release notes (post-hoc edit on GitHub Release page)
- CHANGELOG v35.3.1 entry: "DEPRECATED x25519_generate; will be replaced with proper X25519 in v36.x; do not use for production key exchange"
- No new code shipped

Honest but doesn't add capability. Leaves users with a broken
primitive labeled X25519.

## §3 — Recommendation

**Option A.** Reasoning:
- The bug is real; ignoring it would compound the problem with each
  release
- v35.3.0 just shipped (today); breaking change impact is minimal
  (no users have likely deployed v35.3.0 x25519 yet)
- Adding x25519-dalek is a small dep (~50 KB compiled)
- The "BREAKING" framing is technically accurate but practically
  the v35.3.0 keys weren't usable for real X25519 anyway

**Honest framing for v35.3.1 release notes:**
> "v35.3.0 shipped `x25519_generate` with a broken impl (used
> Ed25519 derivation as proxy per a TODO comment in the source).
> v35.3.1 fixes this with proper x25519-dalek-based implementation
> and adds the `x25519_dh` shared-secret primitive. **Keys
> generated by v35.3.0 will not work with v35.3.1's x25519_dh —
> regenerate keypairs.**"
