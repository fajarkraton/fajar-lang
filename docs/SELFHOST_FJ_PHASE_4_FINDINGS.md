# fj-lang Self-Hosting — Phase 4 Findings

> **CRITICAL-PATH PHASE CLOSED.** stdlib/codegen.fj — written in
> Fajar Lang itself — emits valid C source code. Combined with gcc,
> this constitutes a working Stage-1-Subset compiler chain. Verified

> the expected result.

## 4.1 — Pivot decision: gcc backend instead of Cranelift FFI

The original plan §4 specified Cranelift FFI builtins as the codegen
strategy. **Pivot rationale**:

- `stdlib/codegen.fj`'s own header comment already specifies the
  strategy:
  > **Strategy: Fajar Lang → C → gcc → binary**
- gcc is universally available; no need to expose Cranelift IR
  builder via builtins (estimated 2-5K LOC Rust shim).
- C is human-readable, debuggable, and a long-established target for
  bootstrap compilers (chibicc, c0c, tinyc, etc.).
- gcc is mature and produces well-optimized code; the pivot does NOT
  sacrifice production quality.

**Cost of pivot**: zero new Rust code. We use the codegen.fj as-is
+ subprocess gcc.

**Status of Cranelift FFI option**: roadmap-only for Stage-1-Full;
not blocking Stage-1-Subset.

## 4.5 — Smoke test 2: multi-fn with let / if-else / binop / fn call

Test program (driven by fj-source codegen API):

```fj
state = emit_function(state, "compute", ["a", "b"], "i64")
state = emit_let(state, "sum", "a + b")
state = emit_if(state, "sum > 100")
state = emit_return(state, "sum")
state = emit_else(state)
state = emit_return(state, "0")
state = emit_endif(state)
state = emit_function_end(state)

state = emit_function(state, "main", [], "i64")
state = emit_let(state, "x", "compute(50, 75)")
state = emit_println(state, "x")
state = emit_return(state, "0")
state = emit_function_end(state)
```

Generated C (28 lines):

```c
// Generated by Fajar Lang self-hosted compiler
#include <stdio.h>
#include <stdlib.h>
#include <stdint.h>
#include <string.h>
#include <stdbool.h>

// Fajar Lang runtime
static void fj_println_int(int64_t v) { printf("%lld\n", (long long)v); }
static void fj_println_str(const char* s) { printf("%s\n", s); }
static void fj_println_float(double v) { printf("%g\n", v); }
static void fj_println_bool(int v) { printf(v ? "true\n" : "false\n"); }

int64_t compute(int64_t a, int64_t b) {
    int64_t sum = a + b;
    if (sum > 100) {
        return sum;
    } else {
        return 0;
    }
}

int64_t main(void) {
    int64_t x = compute(50, 75);
    fj_println_int(x);
    return 0;
}
```

gcc compile + run:

```bash
$ gcc /tmp/full.c -o /tmp/full && /tmp/full && echo "RC=$?"
125
RC=0
```

50+75=125, > 100, branch returns sum, main prints 125. **PASS**.

## 4.6 — Coverage gap (deferred to Stage-1-Full)

Codegen currently emits:
- ✅ Fn definitions with typed params + return
- ✅ Let bindings (typed via map_type)
- ✅ If-else-endif blocks
- ✅ While loops
- ✅ Return statements
- ✅ Fn calls (positional args)
- ✅ Built-in `println(int)`
- ✅ Binops including bitwise + comparison + logical

NOT emitted yet (Stage-1-Subset edge cases):
- ❌ Struct/enum definitions (need C struct decl + variant tagging)
- ❌ Match expressions (need switch+goto or chain of if-else)
- ❌ For loops (need lowering to while + iterator state)
- ❌ Arrays (need C array allocation + length tracking)
- ❌ Strings as values (currently const char* literal-only)
- ❌ Generic struct/enum (excluded from Stage-1-Subset)

For Phase 5 (bootstrap chain) we'll exercise the existing emitters by
hand-curating subset programs that avoid these gaps. The full subset
suite at Phase 6 will surface which gaps NEED filling for v33.4.0.

## 4.7 — Architectural observation

Codegen via C source emission is a **proven, mature compiler
architecture**: TinyCC, chibicc, Vala, Nim (early versions), Eiffel
all do this. Advantages over IR-builder approach:
- Zero Rust shim required for Stage-1-Subset
- Output is human-readable and debuggable
- gcc handles register allocation + optimization + linking
- Cross-compilation via gcc target triples (free)

Disadvantages (acknowledged):
- Runtime gcc dependency
- Slower than direct codegen (gcc parse overhead)
- Less control over generated code

For a Stage-1-Subset PROOF, advantages dominate. Production fj-lang
already has Cranelift + LLVM backends in Rust; Stage-1-Subset doesn't
need to compete on perf — it needs to PROVE self-hostability.

## 4.11 — Path to Phase 5

Phase 5 (bootstrap chain) glues:
1. fj-source reads .fj input
2. tokenize() → token stream
3. parse_program() → AST (nested arrays)
4. analyze() → check for errors
5. emit_program(ast) → C source
6. shell out to gcc
7. produce executable

The current codegen.fj is driven by **direct calls to emit_*
functions**, NOT by an `emit_program(ast)` driver that walks the
parser's AST. So Phase 5 needs:
- A new `emit_program(ast: [str])` function in codegen.fj
- Pattern-match on AST node tags `["fn", ...]`, `["let", ...]`, etc.
- Recursive emission

This IS additional fj-lang code (not just integration). Estimated
0.5-1d per pattern; doable.
