Language Comparison

How Fajar Lang compares to Rust, C, and Python for embedded ML and OS workloads.

Feature Fajar Lang Rust C Python
Memory safety (no GC) Yes (ownership) Yes (ownership + lifetimes) No (manual) GC-managed
Lifetime annotations Not required Required N/A N/A
Native tensor type First-class Via libraries No Via NumPy
Compile-time shape checking Yes No No No
Built-in autograd Yes (backward, grad) No No Via PyTorch
OS kernel development Yes (@kernel context) Yes (no_std) Yes No
Context isolation (ML + OS) Compiler-enforced Convention only No No
Inline assembly Yes Yes (asm!) Yes (__asm__) No
Cross-compilation (ARM64, RISC-V) Built-in Via targets Via toolchain Interpreted
INT8 quantization Native Via libraries Manual Via frameworks
Pattern matching Yes Yes No (switch) match (3.10+)
Generics Monomorphized Monomorphized No (macros/void*) Type hints only
Trait / interface system Traits + dyn dispatch Traits No Protocols (ABC)
Async / await Yes Yes No Yes (asyncio)
REPL Yes (fj repl) No (unofficial) No Yes
Package manager Built-in (fj.toml) Cargo No standard pip
Learning curve Moderate Steep Moderate (unsafe) Easy
Note: This comparison reflects the design goals and current capabilities of each language for the specific domain of embedded ML + OS integration. Each language has strengths in other domains not represented here. Rust and C have decades-mature ecosystems; Python dominates data science tooling. Fajar Lang is purpose-built for the intersection of OS-level control and ML inference in a single binary.
